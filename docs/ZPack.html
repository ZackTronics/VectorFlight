VectorFlight Pegasus Protocol Specification

The VectorFlight Pegasus protocol, also known as the ZackPacket Protocol, is a specialized telemetry and control specification designed for high-performance communication between Ground Control Stations (GCS) and flight hardware. It utilizes a custom reliability layer over UDP to ensure consistent synchronization of "SuperVariables" across lossy network links.

Variable Update Frequency & Latency Specification

The update frequency for variables in the VectorFlight Pegasus system is a dynamic, multi-tier architecture. It scales update rates based on data priority, network health, and hardware constraints.

1. The Polling Tiers

Standard telemetry is categorized into polling classes, defined by the pollingInterval[i] settings.

  • Class 1 (High-Priority): Aimed at critical flight dynamics (IMU, Attitude). Typically configured for 20ms – 50ms intervals.
  • Class 2 (Medium-Priority): Used for environmental and status telemetry (Battery, Temperature). Typically configured for 100ms – 500ms intervals.
  • On-Demand (Handshake/Static): Configuration parameters (PID gains) are synchronized during the 10-step handshake or upon explicit request.

2. Event-Driven Immediate Updates

The system does not always wait for a polling cycle. When a user modifies a variable in the Ground Control Station UI, the PleaseXmitChanges() logic triggers an immediate transmission.

  • Delta Detection: The system compares the current value against the Cached3 (last sent) value.
  • Immediate Dispatch: If a difference is detected, a ZPack_WriteSuperVar (0xB3) is queued for the next available transmission window, bypassing the standard polling timers.

3. Network-Aware Throttling

If the network link degrades, the system automatically slows down variable updates to maintain connection integrity.

The xmitInterval stretches based on the number of unacknowledged packets in the sentPacketsLog:

xmitInterval = 37.0 * (packetWindow / (packetWindow - (unackedPacketCount * 1.07)))

In optimal conditions (0 unacked), the system transmits at the maximum rate allowed by the polling settings. If the window (default 3) fills, updates can slow significantly (e.g., from 20ms to 200ms+) until ACKs are received.

4. Controller Latency (XBox Input)

Joystick inputs are treated as the highest priority data stream.

  • Target Rate: Governed by iControllerPacketInterval, typically set to 20ms (50Hz).
  • Crucial Status: Controller words (0x36) are marked as criticalPacket, ensuring they are tracked in the retry log and retransmitted if an ACK is not received within the RTT-based timeout.

2. Binary Framing Specification

Communication is structured into "ZackPackets," which act as containers for one or more atomic "Packlets".

2.1 ZackPacket Header & Footer

Every UDP datagram follows this top-level structure:

Offset (Bytes) Field Type Description
0 Packet Type uint8 Must be 0x25 for ZackPackets.
1 - 2 Sequence Number uint16 Global counter (glbZackPacketSequence) used for tracking and ACKs.
3 - (N-2) Payload Binary A concatenated stream of one or more Packlets.
N-1 - N CRC-X25 uint16 Checksum for the entire packet calculated using the X25 polynomial.

2.2 Packlet Structure

Packlets allow multiple message types (e.g., Heartbeat and Write) to be bundled into a single transmission.

Offset (Bytes) Field Type Description
0 - 1 Packlet Size uint16 Total size of the packlet (Header + Payload).
2 - 3 Message ID uint16 Identifier for the operation (MsgID).
4+ Payload Var Variable-length data specific to the MsgID.

3. Message Directory (MsgIDs)

5. Reliability and State Management

5.1 Retransmission Logic

The protocol uses a selective retransmission strategy for Crucial Packets.

  • Storage: Outgoing crucial packets are archived in sentPacketsLog.
  • Timeout: The system waits for msecsRetyInterval, calculated as $averageRTT \times 7$.
  • Recovery: If no ACK is received, the crucialPacketBody is re-queued for transmission.

5.2 Connection Status Levels

Connection quality is assessed based on the ratio of unacknowledged packets to the packetWindow.

Level Status Condition
0 Disconnected No packets received for 3x Heartbeat interval.
1 Congested Unacked count ≥ 70% of packetWindow.
2 Degraded Unacked count ≥ 30% of packetWindow.
3 Optimal Unacked count < 30% of packetWindow.

5.3 Handshake State Machine

The system follows a strict synchronization flow (Step 0 to 10) to ensure GCS and Aircraft parameter parity.

Step State Action
0 Initial Reset VarHandshakeStatus for all variables.
1 Request Send Read Requests for all plsHandshk variables.
2 Synchronizing Wait for Read Responses and compare with local values.
3 Conflict Check Detect mismatches; trigger user prompt if differences exist.
10 Steady State Begin high-speed telemetry and joystick control.

Advanced Specification: ZackPacket (0x25)

This document provides an exhaustive breakdown of the ZackPacket container architecture, the Packet Window congestion logic, and the reliability matrix governing acknowledgments.

1. Detailed ZackPacket (ZPack) Anatomy

The ZackPacket is a high-level container designed to encapsulate multiple "Packlets" within a single UDP datagram to minimize header overhead and maximize throughput.

1.1 Packet Encapsulation Layout

Component Sub-Component Type Bytes Description

Sequence Number unsigned short 2 Incrementing ID (glbZackPacketSequence) for tracking.
Payload Packlet Stream Binary Var Contains one or more Packlets (detailed in Section 1.2).
Fixed Footer CRC-X25 uint16_t 2 16-bit checksum calculated over the entire packet.

1.2 Packlet Internal Structure

Every message within the ZPack payload must follow the Packlet framing to allow the receiver to "de-stack" multiple messages.

Field Type Bytes Description
Packlet Size uint16_t 2 Total bytes in this Packlet, including these 2 bytes.
Message ID (MsgID) uint16_t 2 The specific command or data type (e.g., 0xB3 for Write).
Message Body Var Var The actual data (e.g., VarID + Value).

2. The Packet Window & Flow Control

The Packet Window is a sliding-window mechanism used to prevent network congestion, particularly on low-resource hardware like the Raspberry Pi.

Window Limitation: The packetWindow is default-set to 3. This small window is intentionally chosen to counteract undesired buffering on the RPI.

2.1 Connection Status Determination

The system calculates unackedPacketCount by checking the sentPacketsLog. This count is compared to the packetWindow to set the connectionStatus.

Status Level Logic Condition Network Behavior
3 (Optimal) unacked < 30% of Window Full telemetry enabled; minimum transmit intervals.
2 (Degraded) 30% ≤ unacked < 70% Telemetry continues; potential slight increase in intervals.
1 (Congested) unacked ≥ 70% of Window System throttles polling; focus on Heartbeat and Criticals.
0 (Timed Out) No RX for 3x Heartbeat Link marked as dead; Handshake reset to Step 0.

2.2 Adaptive Transmit Interval

If the HelicopterMode is inactive (Ground Station mode), the system dynamically stretches the xmitInterval based on the window utilization:

xmitInterval = 37.0 * (packetWindow / (packetWindow - (unackedPacketCount * 1.07)))

3. Acknowledgment (ACK) Matrix

Not all packets are treated equally. The system categorizes messages to ensure reliable delivery of critical data without flooding the network with ACKs for ACK messages.

3.1 Which Messages are Acknowledged?

When a ZPack is received, the system checks each Packlet's MsgID to decide if it should trigger a ZPack_Acknowledge (0xAC).

MsgID Message Type Requires ACK? Reasoning
0xB3 WriteSuperVar YES Ensures the remote system successfully updated the parameter.
0xB4 ReadRequest YES Confirms the request was received before the response is sent.
0xB5 ReadResponse YES Confirms the requested data reached the requester.
0x36 XBoxController YES Critical for flight safety to ensure control inputs are received.
0xB6 HeartBeat YES Used to verify link two-way integrity.
0xAC Acknowledge NO Prevents infinite ACK loops.

3.2 Crucial vs. Non-Crucial Transmission

The releaseOutPacket(int isCrucial) function determines how a packet is tracked.

  • Crucial Packets (1): The originalData is archived in the sentPacketsLog. If the sequence number is not cleared by an incoming ACK, the system retransmits the data.
  • Non-Crucial Packets (0): Sent immediately but not archived. If lost, the system relies on the next periodic update (e.g., standard telemetry polling).

VectorFlight Pegasus: Flow Control & Protocol Specification

This technical specification details the VectorFlight Pegasus communication system, focusing on its sophisticated flow control mechanisms and the ZackPacket (0x25) architecture.

1. Flow Control Logic: Dynamic Transmission Timing

The system does not rely on a fixed transmission rate. Instead, it uses a closed-loop feedback mechanism to adjust the frequency of packets based on network latency and congestion.

1.1 Determining Transmission Frequency

The code uses two primary methods to decide when to transmit:

  • External Timing: A high-level Qt timer (timer2) triggers decideIfToSendPackets().
  • Internal Delta-Time Logic: The PeriodicityFunction tracks the time elapsed since the last occurrence of a specific event using toMSecsSinceEpoch(). If the elapsed time exceeds a defined periodMsecs, the function returns 1, allowing the transmission to proceed.

1.2 Adaptive Congestion Scaling

When the system is in Ground Station mode, it dynamically stretches the interval between transmissions to prevent saturating the link. The xmitInterval is calculated using the following formula:

xmitInterval = 37.0 * ((float) packetWindow / ((float) packetWindow - ((float) unackedPacketCount * 1.07)));

As unackedPacketCount approaches the packetWindow limit, the time between allowed transmissions increases exponentially, ensuring the link has time to recover.

2. The Packet Window & Reliability Layer

The Packet Window serves as a threshold for active, unacknowledged crucial data.

2.1 Window Logic & Performance

Parameter Standard Value Operational Impact
packetWindow 3 Keeps buffers short to prevent latency spikes on the Raspberry Pi.
msecsRetyInterval Adaptive Set based on current averageRTT to optimize retransmission speed.
crucialPacket flag Binary Determines if a packet is stored in the sentPacketsLog for tracking.

2.2 Acknowledgment Matrix

Incoming ZackPackets trigger an Acknowledge packlet (0xAC) only for specific message types to avoid unnecessary network traffic.

Message ID Description Requires ACK
0xB3 WriteSuperVar Yes - Confirms remote parameter update.
0x36 XBoxController Yes - Ensures flight control commands arrive.
0xAC Acknowledge No - ACK-of-ACK is suppressed to prevent loops.

3. ZackPacket (ZPack) Container Specification

Every UDP datagram is framed as a ZackPacket container (ID: 0x25).

3.1 Binary Frame Layout

Offset Field Type Description
0 Type ID uchar Always 0x25.
1-2 Sequence Num uint16 Unique identifier for ACK matching.
3 to N-3 Packlets Binary Variable number of bundled sub-messages.
N-2 to N CRC-X25 uint16 16-bit checksum calculated via bit-wise XOR and shifts.

3.2 SuperVariable Serialization

Data is converted into raw byte streams using specialized num2QByteArray overloads.

  • Integer Types: Manually masked and shifted into Big or Little Endian order based on the isBigEndian flag.
  • Floating Point: Serialized via memcpy for direct IEEE 754 compatibility.
  • Strings: Converted to UTF-8 with a fixed maximum length of 50 characters.

4. State Synchronization: The Handshake

A 10-step state machine governs initial connection and parameter verification.

Handshake Step Phase GCS Protocol Behavior
1 Initiation GCS sends ZPack_ReadRequest for all plsHandshk variables.
2 Verification GCS waits for responses and marks VarHandshakeStatus based on value parity.
3-5 Resolution If mismatches occur, user intervention is required to sync aircraft gains with GCS defaults.
10 Steady State Handshake complete; periodic variable polling begins.

XBox Controller State: Binary Packing Specification

The transmission of controller states in the VectorFlight Pegasus protocol is handled by the ZPack_XBoxControllerWord (MsgID: 0x36). To maximize bandwidth efficiency, the system packs multiple boolean buttons into bit-fields and serializes analog stick data into a 15-byte payload.

1. The 15-Byte Payload Layout

The controller state is packed into a QByteArray titled ctrlWord within the push_ZPack_XBoxController function.

Byte Offset Field Type Packing Method
0 Buttons (Set 1) Bit-mask A, B, X, Y, Back, Start, Guide, Left Joy (LJ) click.
1 Buttons (Set 2) Bit-mask Right Joy (RJ) click, Left Bumper (LB), Right Bumper (RB).
2 D-Pad (Hat) Nibbles KPD_X and KPD_Y mapped to lower and upper nibbles.
3 - 4 Left Trigger (LT) uint16 Serialized via num2QByteArray.
5 - 6 Right Trigger (RT) uint16 Serialized via num2QByteArray.
7 - 8 Left Joy Mag uint16 Magnitude of the left joystick vector.
9 - 10 Left Joy Ang uint16 Angle of the left joystick vector.
11 - 12 Right Joy Mag uint16 Magnitude of the right joystick vector.
13 - 14 Right Joy Ang uint16 Angle of the right joystick vector.

2. Bit-Field Mapping Details

Boolean states are manually shifted into 1-byte buffers before being appended to the packet.

2.1 Byte 0: Button Mask A

Bit Controller Button Logic (C++)
7 (0x80) A Button 0x80 & (0xFF * (Ctrl.A && 1))
6 (0x40) B Button 0x40 & (0xFF * (Ctrl.B && 1))
5 (0x20) X Button 0x20 & (0xFF * (Ctrl.X && 1))
4 (0x10) Y Button 0x10 & (0xFF * (Ctrl.Y && 1))
3 (0x08) Back Button 0x08 & (0xFF * (Ctrl.Back && 1))
2 (0x04) Start Button 0x04 & (0xFF * (Ctrl.Start && 1))
1 (0x02) Guide Button 0x02 & (0xFF * (Ctrl.Guide && 1))
0 (0x01) LJ Click 0x01 & (0xFF * (Ctrl.LJ && 1))

2.2 Byte 2: D-Pad / Keypad State

The D-Pad uses 4 bits per axis to represent direction.

  • Lower Nibble (Bits 0-3): Stores Ctrl.KPD_X.
  • Upper Nibble (Bits 4-7): Stores Ctrl.KPD_Y shifted left by 4.

3. Analog Serialization Logic

Analog sticks and triggers are converted from the Struct_XBox360Controller into 2-byte uint16 segments.

Polar Coordinate Conversion: Unlike standard HID drivers that send X/Y coordinates, this protocol packs Magnitude and Angle. This is likely optimized for flight physics calculations involving thrust vectors.
  • Serialization: Uses num2QByteArray(unsigned short int X) which performs bitwise masking to split the 16-bit value into two 8-bit characters.
  • Endianness: Adheres to the isBigEndian flag, appending bytes in forward or reverse order based on the system configuration.

4. Unpacking and Receiver Validation

The aircraft or receiving computer performs the inverse operation in process_ZPack_XBoxControllerWord.

  • Sequence Validation: Rejects packets that are "excessively old" or out of order by comparing incomingPacketSeqNum to highestReceived.
  • Bit Extraction: Uses bitwise AND operations (e.g., 0x80 & (unsigned char)QByteArray2lluint(...)) to reconstruct the boolean button states.
  • Hardware Injection: On the Raspberry Pi, these values are immediately written to the flight board via I2CSuperVarWrite.

ZPack to SuperVariable Mapping Specification

This document details the internal architectural link between the ZackPacket (ZPack) binary protocol and the SuperVariable (Vars) memory management system. The mapping allows the communication system to resolve unique 16-bit IDs into direct memory pointers for real-time telemetry and control.

1. The Mapping Bridge: VarID to Pointer

Every SuperVariable is assigned a unique varIDtype (unsigned short). The system uses a multi-stage lookup to resolve this ID into a specific index within the Vars structure.

1.1 Lookup Logic

The core resolution function is getTypeAndArrIndex(varID, index, type). It performs the following steps:

  • Static Table Check: It first checks a static lookup table (cache) to see if the mapping was previously resolved.
  • Type-Array Iteration: If not cached, it iterates through all registered variable arrays (uint8, float32, etc.) until a matching ID is found.
  • Metadata Retrieval: Once the matching array and index are found, the system retrieves the actual memory pointer from the corresponding Ptr_xxx array.

2. Data Serialization Mapping

When a variable is transmitted or received, it is mapped from its native C++ type to a serialized QByteArray.

Packet Payload Component C++ Data Type SuperVar Pointer Array Serialization Logic
Body Bytes 0-1 varIDtype Vars.ID_xxx Looked up via getVarID.
Body Bytes 2+ (Type 1) unsigned char *Vars.Ptr_uint8[i] Direct byte copy.
Body Bytes 2+ (Type 7) float *Vars.Ptr_float32[i] IEEE 754 memcpy.
Body Bytes 2+ (Type 9) QString *Vars.Ptr_QString[i] fromUtf8 / toUtf8.

3. Operational Data Flow

3.1 Reception Flow (ZPack → SuperVar)

When a ZPack_WriteSuperVar (0xB3) arrives, the system updates the local application state:

  1. The packet is de-framed to extract the VarID and the RemoteVal (raw bytes).
  2. The system calls setValue(VarID, RemoteVal).
  3. Inside setValue, the RemoteVal is cast back to the native type using QByteArray2lluint or QByteArray2float.
  4. The resulting value is written directly to the memory address held by the pointer in Vars.Ptr_xxx[index].

3.2 Transmission Flow (SuperVar → ZPack)

When a variable needs to be sent (e.g., via pushSuperVarWrite), the reverse occurs:

  1. The system calls getValAsQByteArray(VarID).
  2. The function dereferences the pointer in Vars.Ptr_xxx[index] to get the live application value.
  3. This value is serialized into a QByteArray using the appropriate num2QByteArray overload.
  4. The serialized bytes are appended to the packlet body following the 2-byte VarID.
Direct Memory Interaction: Because the system uses pointers (Ptr_xxx), any update to the variable by the application logic (e.g., autopilot calculating a new position) is immediately available for the next transmission without an explicit copy step.

4. Variable Metadata Mapping

Beyond the raw value, the ZPack logic relies on metadata stored in the Vars structure to govern behavior.

Metadata Field Logic Impact Reference
retainFlags_xxx Determines if the variable is included in the Handshake (plsHandshk) or saved to disk (defIsRetain).
pollRate_xxx Determines which pollingInterval (Fast vs. Slow) the variable belongs to.
BoardAddress_xxx Maps the SuperVariable to a specific register offset in the 2048-byte I2C Flightboard buffer.
